# Implementation Guide: Set Up Database (PostgreSQL + Prisma)

## Overview

Define the complete Prisma schema for Phase One â€” all models, enums, relations, and performance indexes. This is the data foundation that every API endpoint and feature depends on. PostgreSQL and Prisma are already initialized in `apps/api`.

## Prerequisites

- PostgreSQL installed and running locally (Mac dev environment)
- Prisma initialized in `apps/api` with PostgreSQL provider
- `DATABASE_URL` configured in `.env`

## Scope

Prisma schema file in `apps/api` â€” defining all models, enums, relations, and indexes. No application code, no API routes, no seed scripts in this step.

## Architecture Context

All models are defined in `db_schema.md`. The schema supports:

- **User management** with role-based access (Admin/Viewer)
- **Time-series metrics** collected every 15 seconds (System, Docker, Node.js)
- **Activity logging** tied to users
- **Notifications** with read/unread state
- **System logs** from multiple sources

Key architectural decisions that affect schema design:

- UUIDs for all IDs (portability)
- UTC timestamps everywhere
- 14-day data retention (indexes must support efficient time-range deletes)
- JWT auth with HTTP-only cookies (no session table needed)
- Access code signup flow (AccessCode model â€” from Decisions.md but not yet in db_schema.md)

## Implementation Steps

### Step 1: Define Enums

- **What**: Create the `Role` enum (`ADMIN`, `VIEWER`)
- **Where**: `apps/api/prisma/schema.prisma`
- **Details**: Only one enum needed for Phase One. Keep it simple â€” no custom string mappings needed.
- **Depends on**: None

### Step 2: Implement Core Models

- **What**: Add all 7 models from `db_schema.md`:
  - `User` â€” username/password auth, role assignment
  - `ActivityLog` â€” user actions (LOGIN, PAGE_VIEW), linked to User
  - `SystemMetric` â€” CPU/memory/network with type discriminator
  - `DockerMetric` â€” per-container resource usage
  - `NodeServerMetric` â€” per-process Node.js server stats
  - `Notification` â€” system events with severity and read state
  - `SystemLog` â€” multi-source log entries
- **Where**: `apps/api/prisma/schema.prisma`
- **Details**:
  - Follow the exact field definitions in `db_schema.md`
  - All IDs: `String @id @default(uuid())`
  - All timestamps: `DateTime @default(now())`
  - User.password is stored hashed (bcrypt) â€” schema just stores the string
  - ActivityLog has a relation to User (`userId` foreign key)
  - User has a reverse relation `activityLogs ActivityLog[]`
- **Depends on**: Step 1 (Role enum must exist for User model)

### Step 3: Add AccessCode Model

- **What**: Add the `AccessCode` model for the access-code signup flow
- **Where**: `apps/api/prisma/schema.prisma`
- **Details**: This model was decided in Decisions.md (2026-02-04) but is not yet in `db_schema.md`. Suggested fields:
  ```
  model AccessCode {
    id        String   @id @default(uuid())
    code      String   @unique
    email     String
    role      Role     @default(VIEWER)
    used      Boolean  @default(false)
    expiresAt DateTime
    createdAt DateTime @default(now())
  }
  ```

  - `code`: unique access code string (generated by admin)
  - `email`: tied to a specific email for validation
  - `role`: what role the user gets on signup (default VIEWER)
  - `used`: prevents code reuse
  - `expiresAt`: code expiration
- **Depends on**: Step 1 (Role enum)

### Step 4: Add Performance Indexes

- **What**: Define indexes optimized for the API query patterns
- **Where**: `apps/api/prisma/schema.prisma` (as `@@index` directives on each model)
- **Details**: Based on the API specification, these queries drive index design:

  **SystemMetric**:
  - `GET /dashboard/metrics?type=cpu&period=24h` â†’ query by type + timestamp range
  - Retention cleanup â†’ delete by timestamp
  - Indexes: `@@index([type, timestamp])`, `@@index([timestamp])`

  **DockerMetric**:
  - `GET /docker/containers/:id/metrics?period=24h` â†’ query by containerId + timestamp range
  - Indexes: `@@index([containerId, timestamp])`, `@@index([timestamp])`

  **NodeServerMetric**:
  - `GET /nodejs/servers/:pid/metrics?period=24h` â†’ query by processId + timestamp range
  - Indexes: `@@index([processId, timestamp])`, `@@index([timestamp])`

  **ActivityLog**:
  - Queried by userId and timestamp
  - Indexes: `@@index([userId, timestamp])`, `@@index([timestamp])`

  **Notification**:
  - `GET /notifications?read=false` â†’ query by read status + timestamp (for ordering)
  - Indexes: `@@index([read, timestamp])`

  **SystemLog**:
  - `GET /logs/system?level=error` â†’ query by level + timestamp
  - `GET /logs/application?source=nodejs` â†’ query by source + timestamp
  - Indexes: `@@index([level, timestamp])`, `@@index([source, timestamp])`

  **AccessCode**:
  - Lookup by code (already covered by `@unique`)
  - Indexes: `@@index([email])`

- **Depends on**: Steps 2, 3

### Step 5: Validate Schema

- **What**: Run `npx prisma validate` to verify the schema is correct
- **Where**: `apps/api/`
- **Details**: Catches syntax errors, invalid relations, and type mismatches before migration. Do NOT run `prisma migrate` in this step â€” migration is a separate follow-up task.
- **Depends on**: Step 4

### Parallel Work

- **Can run in parallel**: Steps 2 and 3 (independent models)
- **Sequential chain**: Step 1 (enums) -> Steps 2+3 (models) -> Step 4 (indexes) -> Step 5 (validate)
