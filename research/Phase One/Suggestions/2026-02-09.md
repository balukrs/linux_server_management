## Resolve First

> Decisions that should be addressed before backend development can begin. These are all listed in `Decisions.md` and block the entire backend stack.

### 1. Database Engine (PostgreSQL vs MySQL)

- **Why**: Cannot set up Prisma, create migrations, or define the actual schema without choosing a database
- **Options**: PostgreSQL (better JSON support, more features) vs MySQL (simpler, lighter on resources)
- **Recommendation**: PostgreSQL — better suited for time-series metric data, JSON support useful for flexible log/notification storage, and the architecture diagram already leans toward it
- **Blocks**: Database setup, all API endpoints, data persistence

### 2. Authentication Method (JWT vs Sessions)

- **Why**: Auth is the gateway to every protected feature; API middleware, token handling, and frontend auth state all depend on this
- **Options**: JWT (stateless, refresh token strategy needed) vs Sessions (server-side, simpler revocation)
- **Recommendation**: JWT with refresh tokens — aligns well with the REST API architecture, stateless backend, and the frontend is already a SPA
- **Blocks**: Auth API endpoints, auth middleware, frontend auth integration, all protected routes

### 3. Real-time Data Strategy (WebSocket vs Polling)

- **Why**: Determines how the frontend receives live metric updates; affects both frontend architecture and backend implementation
- **Options**: WebSocket (true real-time, persistent connection) vs Polling (simpler, slight delay)
- **Recommendation**: Start with Polling for Phase One (simpler, sufficient for monitoring), upgrade to WebSocket in Phase Two if needed
- **Blocks**: Dashboard live data, frontend data fetching patterns, backend data push mechanism

### 4. Metric Collection Interval (5s / 15s / 30s / 1min)

- **Why**: Affects database storage requirements, CPU overhead on Raspberry Pi, and chart granularity
- **Recommendation**: 30s for Phase One — good balance for a Raspberry Pi's resources while providing useful chart data
- **Blocks**: Backend collector service implementation, storage planning

### 5. Data Retention Period (7 / 14 / 30 days)

- **Why**: Determines database cleanup strategy and storage requirements
- **Recommendation**: 14 days — reasonable history for monitoring without excessive storage on the Pi
- **Blocks**: Cleanup job implementation, storage calculations
