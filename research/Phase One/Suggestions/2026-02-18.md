### 1. System Metric Collection Service

- **Priority**: High
- **Rationale**: Core Phase One feature — the data pipeline that feeds the Dashboard and all historical charts. The backend must collect system metrics from the Raspberry Pi every 15 seconds and persist to PostgreSQL. Without this, no monitoring page can display real data. This is the single highest-impact task remaining.
- **Depends on**: Nothing — database, Prisma models, and shared types are all ready
- **Scope**: Background service in `apps/api` that reads CPU, memory, disk, network metrics from Linux system interfaces (`/proc`, system commands) and stores in SystemMetric table at 15-second intervals.
- **Suggested subtasks**:
  - Create Linux system interface module (read `/proc/stat`, `/proc/meminfo`, `/proc/net/dev`, `df`)
  - Implement CPU metric collection (overall usage %, per-interval delta calculation)
  - Implement memory metric collection (total, used, available)
  - Implement network metric collection (bytes in/out per interface, calculate rate)
  - Implement disk/storage mount collection (per-mount usage via `df` or `/proc/mounts`)
  - Create metric scheduler (15-second interval, persist to SystemMetric table with timestamps)
