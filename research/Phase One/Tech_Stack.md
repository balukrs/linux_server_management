# Server Management System – Tech Stack (Phase One)

## DOCUMENT PURPOSE

This document defines:

- The technology stack used in Phase One
- The responsibility boundaries of each technology
- Explicit exclusions to avoid incorrect assumptions

This document is intended for:

- Human developers
- Automated AI agents

UI/UX design details are intentionally excluded and will be defined in a separate document.

---

## STACK PHILOSOPHY

- Simple
- Explicit
- Maintainable
- Suitable for a single Raspberry Pi
- Optimized for read-only monitoring

Phase One prioritizes **observability and safety**, not control or automation.

---

## OVERALL ARCHITECTURE (PHASE ONE)

- Backend API server
- Frontend web client
- Relational database
- Single-machine deployment
- No distributed components

---

## LANGUAGE CHOICE (PHASE ONE)

### TypeScript (End-to-End)

- Backend: TypeScript
- Frontend: TypeScript

Reasons:

- Type safety across the system
- Clear contracts between backend and frontend
- Improved tooling for humans and AI agents
- Reduced runtime errors

JavaScript-only code is NOT used in Phase One.

---

## BACKEND STACK (PHASE ONE)

### Runtime

- Node.js

Responsibilities:

- System data collection
- API layer
- Authentication
- Authorization
- Notification generation
- Database access

---

### Backend Framework

- Express.js

Responsibilities:

- HTTP server
- REST API routing
- Middleware handling
- Authentication and authorization enforcement
- Input validation

Express.js is used as a **minimal API framework**.

---

### ORM / Database Access

- Prisma ORM

Responsibilities:

- Database schema definition
- Type-safe database queries
- Migrations
- Data persistence

Prisma is the **only** database access layer.

---

### Database Engine

- PostgreSQL

Rules:

- Database runs locally on the same machine
- Selected for relational integrity, JSON support, and Prisma compatibility

---

### Persisted Data (Phase One)

The database stores:

- Users
- Roles
- User activity logs
- System metrics (time-series, limited retention)
- Docker metrics
- Node.js server metrics
- Notifications
- System logs
- Application logs

---

### System Data Collection

- Native Linux system interfaces
- Standard Linux utilities
- Lightweight Node.js system libraries

Responsibilities:

- CPU, memory, and network metrics
- Process listing (read-only)
- Service status (read-only)
- Docker statistics (read-only)
- Storage and mount information (local + NAS)

Explicitly NOT responsible for:

- Process control
- Service control
- Docker lifecycle actions
- System configuration changes

---

## FRONTEND STACK (PHASE ONE)

### Framework

- React

### Build Tool

- Vite

Frontend responsibilities (high-level only):

- Consume backend APIs
- Render monitoring data
- Handle user authentication state

UI structure, layout, styling, and interaction patterns are **out of scope** for this document and will be defined separately.

---

## AUTHENTICATION & AUTHORIZATION (PHASE ONE)

- Authentication handled by backend (Express)
- Role-based access control:
  - Admin
  - Viewer
- All roles are read-only in Phase One

Backend enforces authorization rules.
Frontend reflects permissions but does not enforce security.

---

## NOTIFICATIONS (PHASE ONE)

- Simple, informational notifications
- Generated by backend
- Persisted in database via Prisma
- Exposed through backend APIs

Notifications:

- Do NOT trigger actions
- Do NOT escalate
- Do NOT use rule engines

---

## DEPLOYMENT (PHASE ONE)

### Target Environment

- Single Raspberry Pi
- Linux-based operating system

Deployment characteristics:

- Backend, frontend, and database run on the same machine
- No cloud dependency
- No container orchestration required

---

## EXPLICIT EXCLUSIONS (PHASE ONE)

The following technologies and patterns MUST NOT be assumed:

- Kubernetes
- Docker orchestration or control
- Message queues
- Background job workers
- Prometheus / Grafana
- Infrastructure-as-code tools
- Cloud-managed services
- Serverless platforms

---

## PHASE TWO – POSSIBLE STACK EXTENSIONS (PLANNING ONLY)

Potential future additions:

- Background workers
- Advanced alerting systems
- Long-term metrics storage
- Fine-grained permission models
- Controlled system actions

None of these are part of Phase One.

---

## GUIDING PRINCIPLE

> Explicit scope beats clever abstractions.

END OF DOCUMENT
